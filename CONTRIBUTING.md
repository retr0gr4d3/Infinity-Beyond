# Contributing

This guide explains how a feature is wired through the codebase end to end, using
the **Fake Dev** window as a complete worked example. If you can follow Fake Dev,
you can add any tool window. Read this once, then use the
[checklist](#recreate-fake-dev-from-scratch) at the bottom.

---

## The big picture

A feature almost always spans the same five layers:

```
 View (.axaml)                 What the user sees and clicks
   │  binds commands/properties
 View-model partial (.cs)      Per-window state + commands  (MainWindowViewModel.<Feature>.cs)
   │  SendCommand / SetSetting
 Beyond/Launcher/ModConnection.cs              Serializes JSON, writes the named pipe
   │  pipe: { "Type": ... }
 Beyond/BeyondAgent/TestMod.cs      ProcessLauncherCommands() handles the message in-game
   │  mutates Entity.mainPlayer etc., then SendStatusUpdate()
 OnStatusReceived (core VM)    Mirrors the game's pushed state back into the view-model
```

The launcher and the mod never share memory — they exchange newline-delimited JSON
over a per-session named pipe ([`ModConnection`](Beyond/Launcher/ModConnection.cs) on the
launcher side, `LauncherServer` on the mod side).

### Two kinds of mod interaction

Pick the one that matches your feature:

| Pattern | Use it for | Launcher sends | Mod handles |
|---------|-----------|----------------|-------------|
| **Setting** | A mirrored value/toggle the mod stores (e.g. `helmSpoofActive`) | `SetSetting(name, value)` → `{ "Type":"SetSetting", "Name":…, "Value":… }` | a `case` in the `SetSetting` switch; value is echoed back in `SendStatusUpdate()` |
| **Command** | A one-shot action whose truth lives in the game (e.g. "set my access level") | `SendCommand(type, params)` → `{ "Type":<type>, … }` | an `else if (type == …)` branch that does the work, then calls `SendStatusUpdate()` |

Settings are two-way mirrored (changing the property pushes to the mod; the mod's
status pushes back). Commands are fire-and-forget actions; the resulting state
comes back through the status snapshot. **Fake Dev uses commands** because the real
values live on `Entity.mainPlayer` in the game.

## How BeyondAgent Ties In (The Lifecycle)

To understand how `BeyondAgent` runs inside the game client, here is the complete sequence of compilation, injection, and execution:

### 1. Compilation & Deployment
When you build the solution (via `build.bat` or `dotnet build`), `BeyondAgent.csproj` targets the game's managed assemblies to compile `BeyondAgent.dll`.
A custom MSBuild `PostBuild` target in [`Beyond/BeyondAgent/BeyondAgent.csproj`](Beyond/BeyondAgent/BeyondAgent.csproj) automatically copies the compiled library and `0Harmony.dll` directly into the game's data folder (`<InstallDir>\*_Data\Managed\`).

### 2. Bootstrapping & Assembly Injection
When you click **Launch** in the Launcher, it initializes the session. Before launching the game client process:
1. The Launcher invokes [`AssemblyPatcher.cs`](Beyond/Launcher/AssemblyPatcher.cs).
2. It opens the game's original `Assembly-CSharp.dll` (saving a `.dll.bak` backup if it's the first time patching).
3. Using **Mono.Cecil**, it patches the `AEC.Start()` method (which runs when the game's core engine initializes) to insert a call to:
   ```csharp
   Infinity_TestMod.BeyondLifecycle.Create();
   ```
4. It writes the patched assembly back to the game's `Managed` directory.

### 3. Startup & IPC Initialization
When the game client executes:
1. Unity loads the patched `Assembly-CSharp.dll` and calls `AEC.Start()`.
2. This invokes `BeyondLifecycle.Create()`, loading `BeyondAgent.dll` into the game's Mono runtime.
3. [`BeyondLifecycle.cs`](Beyond/BeyondAgent/Util/BeyondLifecycle.cs) spawns a persistent Unity `GameObject` named `BeyondAgent`, marks it as `DontDestroyOnLoad`, and attaches the `BeyondLifecycle` component to listen to game ticks.
4. The lifecycle component invokes `TestMod.Initialize()`, which reads the unique named pipe ID generated by the Launcher (passed via the environment variable `BEYOND_PIPE`).
5. It spins up a background thread running the pipe server [`LauncherServer`](Beyond/BeyondAgent/Util/LauncherServer.cs) to wait for a connection from the Launcher.
6. The Launcher connects to this named pipe, establishing the IPC channel.

### 4. Hooking Game Events (Harmony & Update Loop)
With `BeyondAgent` running inside the game process:
- **Game Ticks**: The `BeyondLifecycle` component forwards Unity `Update()`, `OnGUI()`, and `OnApplicationQuit()` events to the main [`TestMod`](Beyond/BeyondAgent/TestMod.cs) instance.
- **Harmony Patches**: Patch files inside [`Beyond/BeyondAgent/Patches/`](Beyond/BeyondAgent/Patches/) intercept core game methods (e.g., packet handlers, login forms, camera zoom multipliers, asset bundle loaders) to redirect behavior or capture events.
- **Command Loop**: Every frame, the mod processes any pending commands from the named pipe in `ProcessLauncherCommands()` and replies with `SendStatusUpdate()`.

## Worked example: the Fake Dev feature

Fake Dev lets you spoof your membership and access level and open the game's dev
window. Here is every file it touches and how each tie is made.

### 1. The view — `Beyond/Launcher/Views/FakeDevWindow.axaml`

A normal Avalonia `Window`. Its controls bind to commands and properties on the
view-model (the window's `DataContext`, set when it's opened):

```xml
<TextBlock Text="{Binding MembershipText}" .../>                                  <!-- read state -->
<Button Content="Toggle Membership" Command="{Binding ToggleMembershipCommand}"/> <!-- run a command -->
<Button Content="100" Command="{Binding SetAccessLevelCommand}" CommandParameter="100"/>
<Button Content="Open Game Dev UI" Command="{Binding OpenDevUICommand}"/>
<Button Content="Reset to Default" Command="{Binding ResetFakeDevCommand}"/>
```

Note the naming: a `[RelayCommand]` method `ToggleMembership()` generates a bindable
`ToggleMembershipCommand`. A method parameter (`SetAccessLevel(string levelStr)`)
is supplied by `CommandParameter`.

The code-behind [`FakeDevWindow.axaml.cs`](Beyond/Launcher/Views/FakeDevWindow.axaml.cs) is
empty boilerplate — just `InitializeComponent()`. All logic lives in the view-model.

### 2. The view-model — `Beyond/Launcher/ViewModels/MainWindowViewModel.FakeDev.cs`

This is the "main code" for the feature. `MainWindowViewModel` is **one class split
across partial files**, one per window; this file holds only Fake Dev's state and
commands:

```csharp
public partial class MainWindowViewModel
{
    [ObservableProperty] private int _playerAccessLevel;
    [ObservableProperty] private int _playerUpgradeDays;

    [ObservableProperty]
    [NotifyPropertyChangedFor(nameof(MembershipText))]
    private bool _isMember;

    public string MembershipText => IsMember ? "Active Member" : "Non-Member";

    [RelayCommand]
    private void SetAccessLevel(string levelStr)
    {
        if (int.TryParse(levelStr, out int level))
            _connection.SendCommand("SetAccessLevel", new JObject { ["Level"] = level });
    }

    [RelayCommand] private void ToggleMembership()
        => _connection.SendCommand("SetMembership", new JObject { ["IsMember"] = !IsMember });

    [RelayCommand] private void OpenDevUI()    => _connection.SendCommand("OpenDevUI", null);
    [RelayCommand] private void ResetFakeDev() => _connection.SendCommand("ResetFakeDev", null);
}
```

- `_connection` (the `ModConnection`) and the `_isUpdatingFromMod` guard live in the
  **core** file [`MainWindowViewModel.cs`](Beyond/Launcher/ViewModels/MainWindowViewModel.cs);
  because every partial is the same class, this file can use them directly.
- `[ObservableProperty]`/`[RelayCommand]` come from CommunityToolkit.Mvvm and
  require the class to be `partial`.

### 3. Launching the window — core view-model + `SessionView.axaml`

The window is a singleton per session. The opener lives in the core file:

```csharp
[RelayCommand] private void OpenFakeDevWindow() => ShowToolWindow<FakeDevWindow>();
```

`ShowToolWindow<T>()` ([MainWindowViewModel.cs](Beyond/Launcher/ViewModels/MainWindowViewModel.cs))
re-focuses an open instance or creates one with `DataContext = this` and caches it
until closed. The button that triggers it is in
[`SessionView.axaml`](Beyond/Launcher/Views/SessionView.axaml):

```xml
<Button Content="Fake Dev Options" Command="{Binding OpenFakeDevWindowCommand}" .../>
```

That `DataContext = this` is the crucial tie: the Fake Dev window and the session's
control panel share **the same view-model instance**, so they see the same state and
the same `ModConnection`.

### 4. Crossing to the game — `ModConnection`

`SendCommand("SetAccessLevel", { Level = 100 })` builds
`{ "Type":"SetAccessLevel", "Level":100 }`, appends `\n`, and enqueues it on the
outbound pipe queue (non-blocking; see [`ModConnection.cs`](Beyond/Launcher/ModConnection.cs)).
`SetSetting(name, value)` is just sugar over `SendCommand("SetSetting", …)`.

### 5. Handling it in-game — `Beyond/BeyondAgent/TestMod.cs`

Every frame, `OnUpdate()` calls `ProcessLauncherCommands()`, which drains the pipe
and dispatches on `Type`:

```csharp
else if (type == "SetAccessLevel")
{
    int level = (int)cmd["Level"];
    if (Entity.mainPlayer != null)
    {
        Entity.mainPlayer.AccessLevel = level;
        Entity.mainPlayer.updateNameColor();
        SendStatusUpdate();          // ← push the new truth back to the launcher
    }
}
else if (type == "SetMembership")  { /* Entity.mainPlayer.UpgradeDays = … ; SendStatusUpdate(); */ }
else if (type == "OpenDevUI")      { new DevWindow(new List<string>()).Execute(); }
else if (type == "ResetFakeDev")   { /* restore captured defaults; SendStatusUpdate(); */ }
```

The handlers mutate `Entity.mainPlayer` (the real game object), then call
`SendStatusUpdate()` so the launcher reflects reality.

### 6. The status round-trip

`SendStatusUpdate()` ([TestMod.cs](Beyond/BeyondAgent/TestMod.cs)) packs a snapshot —
including `playerAccessLevel`, `playerUpgradeDays`, `isMember` — into
`{ "Type":"Status", "Settings":{…} }`. Back on the launcher,
`ModConnection.ProcessMessage` raises `StatusReceived`, and the **core**
`OnStatusReceived(JObject settings)` copies the values into the Fake Dev properties:

```csharp
if (settings.TryGetValue("playerAccessLevel", out val)) PlayerAccessLevel = (int)val;
if (settings.TryGetValue("isMember", out val))          IsMember          = (bool)val;
```

This whole method runs inside `_isUpdatingFromMod = true`, so assigning the
properties does **not** echo back to the mod (only user-driven changes do). That is
why mirrored *settings* are safe to push both ways without a feedback loop.

## Recreate Fake Dev from scratch

Suppose Fake Dev did not exist. To add it (or any new tool window), do this:

1. **View** — add `Beyond/Launcher/Views/FooWindow.axaml` (+ trivial `.axaml.cs`). Copy an
   existing tool window for the custom title bar: a `<Window>` with the
   `ExtendClientArea…` / `WindowDecorations="BorderOnly"` attributes, whose root is
   `<DockPanel Classes="WindowRoot"><v:TitleBar DockPanel.Dock="Top" Title="…"/> …`.
   Do **not** redeclare control styles — they come from `App.axaml`. Bind controls to
   `…Command` and property names you will add next.

2. **View-model partial** — add
   `Beyond/Launcher/ViewModels/MainWindowViewModel.Foo.cs` with
   `public partial class MainWindowViewModel { … }`. Put the feature's state
   (`[ObservableProperty]` for mirrored values, or plain props using
   `UpdateSetting(ref …, "name")` for two-way settings) and its `[RelayCommand]`
   methods here. Commands call `_connection.SendCommand(...)` or `SetSetting(...)`.

3. **Launcher** — in the core
   [`MainWindowViewModel.cs`](Beyond/Launcher/ViewModels/MainWindowViewModel.cs) add
   `[RelayCommand] private void OpenFooWindow() => ShowToolWindow<FooWindow>();`, and
   add a button bound to `OpenFooWindowCommand` in
   [`SessionView.axaml`](Beyond/Launcher/Views/SessionView.axaml).

4. **Mod command handler** — in
   [`Beyond/BeyondAgent/TestMod.cs`](Beyond/BeyondAgent/TestMod.cs) `ProcessLauncherCommands()`,
   add an `else if (type == "DoFooThing")` branch (or a `case` in the `SetSetting`
   switch for a mirrored setting). Mutate game state, then call `SendStatusUpdate()`.

5. **Status (only if state must reflect back)** — add your key(s) to
   `SendStatusUpdate()` in the mod, and mirror them in the core
   `OnStatusReceived(...)` on the launcher (inside the `_isUpdatingFromMod` guard,
   which it already is).

6. **Build &amp; verify** —
   ```sh
   dotnet build Beyond.sln -c Release
   ```
   A clean build proves the partials, bindings, and command names line up
   (CommunityToolkit generates the `…Command` properties from your method names).

### Conventions &amp; gotchas

- **One class, many files.** Never create a second view-model for a tool window; add
  a partial file. The shared `_connection` and the single `OnStatusReceived` mirror
  depend on it. (See the saved design note in the repo memory.)
- **Command name ↔ binding name.** `OpenFooWindow()` → bind `OpenFooWindowCommand`.
  Renaming the method renames the generated command — update the XAML binding too.
- **String keys must match on both sides.** The `"Type"` you send, the `Name` in a
  `SetSetting`, and the keys in `SendStatusUpdate()`/`OnStatusReceived` are plain
  strings; a typo silently no-ops. Keep them identical launcher ↔ mod.
- **Theme is centralized.** Shared control styles and the caption-button styles live
  in [`App.axaml`](Beyond/Launcher/App.axaml); the title bar is the reusable
  [`Views/TitleBar.axaml`](Beyond/Launcher/Views/TitleBar.axaml). Window-local `<Window.Styles>`
  is only for genuinely one-off styling.
- **Fixed-size windows.** The custom title bar lives inside the client area, so a
  fixed-height window needs ~30px added to its `Height` versus the old native-chrome
  size.
- **No Hot-Reloading.** The mod does not support dynamic assembly reloading or hot-reloading DLLs. To test new mod patches or agent changes, compile the project and relaunch/restart the game session from the Launcher.
